Universal Link? Dynamic Link?</h3><p name="aa7a" id="aa7a" class="graf graf--p graf-after--h3">最近工作上實在是太常被這些名詞所圍繞，往往搞懂了一種另外一種就又被模糊了，趁著這個時候好好搞清楚各種名詞的差異以及使用情境，順便做個筆記提供給需要的人。</p><blockquote name="2c2c" id="2c2c" class="graf graf--blockquote graf-after--p">先說在 iOS 9 之後，主要以 Universal Link 為主，所以只要先知道 Universal Link 跟 Deep Link 主要的差異在 Universal Link 的長相會是以 https:// 為主，而 Deep Link 的長相會比較像是自定義的 URL Scheme，如果你的手機上剛好有裝 Instagram，你可以在 safari 上打開 instagram:// 就會發現有個提示問你是否要在 Instagram 當中開啟，而 instagram:// 就是 Instagram 自定義的 URL Scheme</blockquote><h3 name="9d9b" id="9d9b" class="graf graf--h3 graf-after--blockquote">Deep Link</h3><p name="9888" id="9888" class="graf graf--p graf-after--h3">翻成中文就是深度連結，可以這樣想，透過與 APP 綁定，而透過 APP 開啟指定的連結，所以會使用 APP 自己所定義的 URL Scheme，如果你的專案裡面有做過類似的設定，那你就會在 info.plist 裡面看到 URL Schemes 這組 key，其所對應的則是該 app 在安裝起來之後，向系統註冊的自定義 URL Scheme，這邊需要注意一個點，因為其是自行向系統來做註冊，回到 Instagram 的例子，如果今天有個人的手機沒有裝 Instagram，又剛好你的 app 註冊的 URL Scheme 是 instagram，而這時候使用者透過 instagram:// 來做開啟時，系統就會開啟你的 app 來做使用。</p><p name="4c3c" id="4c3c" class="graf graf--p graf-after--p">缺點：</p><ol class="postList"><li name="5330" id="5330" class="graf graf--li graf-after--p">URL Scheme 容易衝突</li><li name="cf47" id="cf47" class="graf graf--li graf-after--li">如果使用的 URL Scheme ，不存在清單當中，此時會顯示連結無效</li></ol><p name="09ec" id="09ec" class="graf graf--p graf-after--li"><em class="markup--em markup--p-em">另外提個題外話，為了要避免連結無效，我們可以在 app 內開啟別人的 URL Scheme 之前，先檢查系統是否有此 URL Scheme 來做註冊，而我們如果要做此檢查，則必須在 info.plist 裡面加上一組 key 為 LSApplicationQueriesSchemes 的白名單，裡面須記錄你要詢問的 URL Scheme 不然透過 UIApplication.shared().canOpenUrl 會都拿到 false</em></p><h3 name="27ab" id="27ab" class="graf graf--h3 graf-after--p"><strong class="markup--strong markup--h3-strong">Deferred Deep Link</strong></h3><p name="48e9" id="48e9" class="graf graf--p graf-after--h3">翻成中文就是延遲深度連結，與上面的 Deep Link 大抵相符，主要的差別是可以把所帶入的參數往後來做使用，可以自己客制這個功能，<em class="markup--em markup--p-em">我如果沒記錯的話，iOS 9 之前可以透過 UIPasteBoard 來做存取，之後則是透過 UserActivity 來做使用</em>，但是我現在找不到相關資料，如果有錯誤歡迎指正。也可以採外第三方的 SDK 來直接使用此服務，例如：AppsFlyer / BranchIO 等等</p><h3 name="0c17" id="0c17" class="graf graf--h3 graf-after--p">Universal Link</h3><p name="2372" id="2372" class="graf graf--p graf-after--h3">翻成中文就是通用連結，可以這樣想，因為其與一般的網址共用 https:// 的 protocol，所以如果今天我們訪問的網址，沒有辦法透過 app 來做開啟的時候，就可以直接瀏覽網頁內容，而不會中斷使用者的體驗。</p><p name="c5d8" id="c5d8" class="graf graf--p graf-after--p">其啟用步驟如下</p><ol class="postList"><li name="44ab" id="44ab" class="graf graf--li graf-after--p">創建 apple-app-site-association 檔案</li><li name="7bb7" id="7bb7" class="graf graf--li graf-after--li">上傳到 Server 的 .well-known 根目錄的資料夾中</li><li name="7ce6" id="7ce6" class="graf graf--li graf-after--li">在 App 內啟用 associated-domain 並且加入 applink:{your_domain}</li><li name="9985" id="9985" class="graf graf--li graf-after--li">在 AppDelegate 當中去處理收到 NSUserActivity 的事件</li></ol><p name="49a4" id="49a4" class="graf graf--p graf-after--li">因為這篇內容不想談到實作面，僅想針對名詞與想法來做討論，細節可以參考下列官方文件。</p><div name="f6e9" id="f6e9" class="graf graf--mixtapeEmbed graf-after--p"><a class="graf graf--p graf-after--p">今天如果使用者沒有安裝 APP，其只能開啟網頁，而不能引導使用者前往 App Store 進行下載。所以有些網頁在你載入的時候，會運行一段 javascript ，其會透過 timer 來計時，如果沒有辦法在特定時間內進行開啟 App，就前往 App Store 進行下載。</p><p name="31d9" id="31d9" class="graf graf--p graf-after--p">這樣的做法其實並不準確，而且在特定版本之後，iOS 的 WebView 即使有進行轉導，其 WebView 的 timer 也不會暫停，所以會發生轉到其他 App 後，又到 App Store 去下載的窘境。</p><p name="463d" id="463d" class="graf graf--p graf-after--p">Universal Links 的使用場境僅限 WKWebView / UIWebView / Safari Pages，因此在一些第三方的 WebView 內，可能其有做一些特殊處理，所以會使導頁的功能失效。</p><p name="1c30" id="1c30" class="graf graf--p graf-after--p">而就算可以順利前往 App Store 去進行下載，使用者原先可能要前往特定頁面所帶的參數，在下載 App 開啟後也消失無蹤，會成為資料追蹤上的斷點。</p><h3 name="d433" id="d433" class="graf graf--h3 graf-after--p">Firebase Dynamic Links</h3><p name="05c8" id="05c8" class="graf graf--p graf-after--h3">由 Google 推出的動態連結，其建立在 Universal Link 之上，只是多包了一層服務，解決了上述 Deep Link / Universal Link 所提到的問題。</p><ol class="postList"><li name="5ccb" id="5ccb" class="graf graf--li graf-after--p">針對未安裝 App 的情境，我們可以選擇前往下載頁面或是瀏覽網站</li><li name="75a0" id="75a0" class="graf graf--li graf-after--li">若是透過連結進行導下載，其會透過 Firebase 的機制將原先點擊時所傳入的參數，在我們首次開起 App 後一並帶入，解決資料斷點</li><li name="fbeb" id="fbeb" class="graf graf--li graf-after--li">客製化一頁 Preview Page，並且在這頁可以上傳圖片</li></ol><p name="8d92" id="8d92" class="graf graf--p graf-after--li">雖然是建立在 Universal Link 之上，可是其一並解決了，開網頁與導下載的需求，以及實作 Deferred Deep Link 可以把參數往後帶，優化了使用者的體驗與增加轉換率，最後甚至可以透過 Firebase 後台來觀測此連結的使用數據與下載的轉換率等等。</p><p name="d8e8" id="d8e8" class="graf graf--p graf-after--p"><a

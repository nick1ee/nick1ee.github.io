裡面的資料筆數，以及所要顯示的資料內容。</p><p name="4983" id="4983" class="graf graf--p graf-after--p">我這邊補充說明一下 MVVMR 的 R，其實 MVVMR 跟 MVVM 的概念大抵相同，只是我們常常在程式碼內生成下一個 View Controller 並且透過 Navigation Controller Push / Pop graf-after--p">這時問題就來了，誰該來做這件事？ View? ViewModel? Model?</p><figure name="6489" id="6489" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*2R7h7yh5jXMJop11wBvUZQ.png" data-width="500" data-height="349" src="https://cdn-images-1.medium.com/max/800/1*2R7h7yh5jXMJop11wBvUZQ.png"></figure><p name="54a3" id="54a3" class="graf graf--p graf-after--figure">Router 這時就派上用場了，我們可以在 Router 中依據邏輯判斷，決定螢幕接下來往哪走，然後在 Router 中把畫面生成，並且使用 Navigation Controller.Push 前往下一個畫面。</p><p name="2cc2" id="2cc2" class="graf graf--p graf-after--p graf--trailing">如此就可以降低耦合度，而且也不會讓你們的 View Controller 彼此間互相擁有。</p></div></div></section><section name="d1c1" class="section section--body"><div class="section-divider"><hr class="section-divider"></div><div class="section-content"><div class="section-inner sectionLayout--insetColumn"><h3 name="0dc7" id="0dc7" class="graf graf--h3 graf--leading">總結</h3><p name="3de2" id="3de2" class="graf graf--p graf-after--h3 graf--trailing">大家在撰寫程式碼的時候，只要記得時時問問自己，這個方法寫在這裡對嗎？現在在寫的這個是 Model? <section name="7793" class="section section--body section--first"><div class="section-divider"><hr class="section-divider"></div><div class="section-content"><div class="section-inner sectionLayout--insetColumn"><h3 name="a46b" id="a46b" class="graf graf--h3 graf--leading graf--title">Software Design Architecture — MVP</h3><p name="b49e" id="b49e" class="graf graf--p graf-after--h3">大家好，繼上一篇文章談論到 MVVM 之後，想要趁著這一波的熱潮，一並把 MVX 系列家族一起講完，這次要跟大家聊聊的主題是 MVP !</p><p name="801e" id="801e" class="graf graf--p graf-after--p">在 MVP 當中，就像我們在前一篇 MVVM 當中所提到的，View 依然是擔任被動的角色，只負責處理顯示資料，除了與資料有關的事情，其他都不會在這一個角色上負責。而 Model 的角色跟其他 MVX 家族一樣，負責做一些資料或商業邏輯的處理。</p><figure name="8be0" id="8be0" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*PgZD4BwPo-vxgP0Rp_BEGg.png" data-width="638" data-height="359" src="https://cdn-images-1.medium.com/max/800/1*PgZD4BwPo-vxgP0Rp_BEGg.png"></figure><p name="263c" id="263c" class="graf graf--p graf-after--figure">主要的差別部分在於 Presenter ，何謂 Presenter？</p><p name="3267" id="3267" class="graf graf--p graf-after--p">我這邊自己的想法是，他其實是跟 ViewModel 一樣的角色，處理一樣的事情。只是有幾個差別的地方而已，讓我對你慢慢道來…</p><figure name="4a33" id="4a33" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*Tpgzmn7njb0GE3i7zldXkQ.png" data-width="900" data-height="515" src="https://cdn-images-1.medium.com/max/800/1*Tpgzmn7njb0GE3i7zldXkQ.png"></figure><p name="a4a5" id="a4a5" class="graf graf--p graf-after--figure">在 MVP 中，Presenter 是不能直接與 View 直接進行溝通的，必須透過 Protocol，很像之前跟大家談過的 Delegate 的概念。</p><p name="09e7" id="09e7" class="graf graf--p graf-after--p">Presenter 有資料，可是他不想自己做顯示，他想找人幫他顯示資料。但不是每個人都可以幫上這個忙，必須要符合 protocol 的人才可以來處理這些事。</p><p name="c4c8" id="c4c8" class="graf graf--p graf-after--p">實際上在處理的時候，我們可能在 protocol 當中宣告一些刷新的方法，然後讓 View Controller Conform 這個 Protocol 之後 ，去實現一些畫面刷新的方法，然後讓 Presenter 透過 delegate 去呼叫這些方法來把資料拿出來做顯示。</p><p name="7413" id="7413" class="graf graf--p graf-after--p graf--trailing">也就是說在 Presenter 當中，會透過一個 delegate 的變數，直接去存取 View 來幫他做事情。所以在 MVP 當中，View 擁有 Presenter，可是 Presenter 也可以透過 Protocol 存取 View。</p></div></div></section><section name="5be6" class="section section--body"><div class="section-divider"><hr class="section-divider"></div><div class="section-content"><div class="section-inner graf-after--p">我自己認知的大概有幾點，拿出來跟大家分享，如果有新的想法也歡迎讓我知道！</p><ol class="postList"><li name="1acd" id="1acd" class="graf graf--li graf-after--p">存取 View 的方法</li></ol><p name="6c8f" id="6c8f" class="graf graf--p graf-after--li">兩者都是 View 擁有 Presenter/ViewModel ，可是在 ViewModel 之中，很單純的就只是做資料的顯示，沒有辦法存取到 View 本身。所以我們自然無法透過 View Model 來做畫面刷新的動作了。</p><p name="e9d2" id="e9d2" class="graf graf--p graf-after--p">但在 MVP 中呢？ 因為可以透過 delegate ，只要在 View 實作 Protocol 的時候，先定義好自己想要做的事情，接著就可以呼叫 delegate 來幫你做事了。</p><p name="6066" id="6066" class="graf graf--p graf-after--p">2. 資料如何更新？</p><p name="b484" name="4ab0" id="4ab0" class="graf graf--p graf-after--pre">但是在 MVVM 中，比較常的會用到 RxSwift 的寫法，這塊我也還沒研究，只知道他讓 string 具有 reference 的方式，也就有點類似 objective-c 現在的處理方式。如此一來當資料發生變動的時候，我們即使沒有寫到下列的程式碼</p><pre name="1257" id="1257" class="graf graf--pre graf-after--p">tableView.reloadData() </pre><p name="99b1" id="99b1" class="graf graf--p graf-after--pre">介面一樣會更新！</p><p name="c07c" id="c07c" class="graf graf--p graf-after--p">但是有沒有辦法在純 swift 的環境下完成這個目的呢？</p><p name="335a" id="335a" class="graf graf--p graf-after--p graf--trailing">有的，可以透過 KVO / Notification 來追蹤物件的變化，一但物件發生變化，就可以把值ㄧ並完成更新了。</p></div></div></section><section name="dcd7" class="section section--body section--last"><div class="section-divider"><hr class="section-divider"></div><div

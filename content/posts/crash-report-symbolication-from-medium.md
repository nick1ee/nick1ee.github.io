<section name="5862" class="section section--body section--first section--last"><div class="section-divider"><hr class="section-divider"></div><div class="section-content"><div class="section-inner sectionLayout--insetColumn"><h3 name="b70c" id="b70c" class="graf graf--h3 graf--leading graf--title">Crash 的路上有你有我</h3><p name="e45e" id="e45e" class="graf graf--p graf-after--h3">寫 iOS 也快兩年了，在開發過程中遇到的 crash 其實並不可怕，從 QA 那邊測到回報的也還好，最可怕的是有出現過，但是卻不知道怎麼重現的 bug，通常這種問題在產品上線之後，都會莫名的大量出現XD</p><p name="484a" id="484a" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">現在可以協助我們分析 Crash 的工具其實很多，包含我們在上傳 ipa 的時候，AppStoreConnect 上面也會有相關的報告，甚至我們自己使用的 Fabric, Crashlytics 等等，都可以協助我們來做分析，但是那是有 debug symbol 的情況下。</em></p><p name="c322" id="c322" class="graf graf--p graf-after--p">如果我們今天是從用戶的手機拿到 Crash Report，沒有 debug symbol 的支援，我們該如何繼續往下，讓我們繼續看下去…</p><p name="b76b" id="b76b" class="graf graf--p graf-after--p">首先，讓我們先了解整份的 crash report 主要分成哪幾個部分</p><blockquote name="fc60" id="fc60" class="graf graf--blockquote graf-after--p">1. APP 資訊<br>當中會提到這份報告所產生的時間，與機型版本等等。<br>需要注意的是有一個欄位是 Version ，其組成為 build+(version)，我們需要用這個欄位來比較出該 build 是在哪一個 commit sha 與 版本的產物，才能順利的拿到 debug symbol 來做解譯。</blockquote><p name="fccd" id="fccd" class="graf graf--p graf-after--blockquote">這邊可以提一下，我們在包 APP 的過程會有一個產出物是 debug symbol (.dSYM)，其會在我們 Archive 的時候自動產生，這個 bundle 當中包含該 app 與包進去的 Libarary 所需要用到的 dSYM 的檔案。</p><p name="f799" id="f799" class="graf graf--p graf-after--p">這邊還需要注意一件事情，如果是拿了別人的 Crash Report 要來進行分析，但是該作業系統並非本機 Xcode 所能支援的版本，這時候可以將該機器連接傳輸線到電腦，讓 Xcode 去進行 processing debug symbol 的動作。</p><figure name="c1ab" id="c1ab" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*kNNLrO_KN_Jf8EHgZuEjNg.png" data-width="2413" data-height="271" src="https://cdn-images-1.medium.com/max/800/1*kNNLrO_KN_Jf8EHgZuEjNg.png"></figure><blockquote name="9a53" id="9a53" class="graf graf--blockquote graf-after--figure">2. Exception 資訊<br>這邊的資訊中可以看到問題發生的型別，錯誤代碼，發生原因等等，最重要的是可以看到有一個欄位是 triggered by Thread ，這邊即表示 crash 發生在哪條 thread 上，如果為 0 就代表是 main thread.</blockquote><figure name="b8ab" id="b8ab" class="graf graf--figure graf-after--blockquote"><img class="graf-image" data-image-id="1*D3MUGmAp7SOuQL_QQDIMVg.png" data-width="426" data-height="100" src="https://cdn-images-1.medium.com/max/800/1*D3MUGmAp7SOuQL_QQDIMVg.png"></figure><p name="b523" id="b523" class="graf graf--p graf-after--figure">3. Thread call stack<br>這邊會清楚了列出在問題發生當下，每一條 thread 的 call stack，stack 本身有堆棧的概念，所以記得這邊閱讀的順序會是從下到上，即你在 A 裡面呼叫了 B 你會看到的結果會是，B 在下 A 在上的 call stack。</p><figure name="4b79" id="4b79" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*v33klPOe8GhC_4HiQfs4RQ.png" data-width="717" data-height="1047" data-is-featured="true" src="https://cdn-images-1.medium.com/max/800/1*v33klPOe8GhC_4HiQfs4RQ.png"></figure><blockquote name="a03d" id="a03d" class="graf graf--blockquote graf-after--figure">4. Binary Images<br>最下方的 Binary 即列出該 APP 載入了哪些 Library 與其所佔用的記憶體位置。這邊需要注意的就是，通常問題都會發生在我們自己專案當中，所以我們要先找到專案的名字那一行，然後可以看到前面的第一個記憶體位置，即為該 Library 被載入時候的記憶體位置，我們稱做 load address 我們稍等需要這個記憶體位置來使用。</blockquote><p name="7a29" id="7a29" class="graf graf--p graf-after--blockquote">此外我們還需要注意到，我們在載入 App 的 Library 的時候，是哪個平台，ARM64/ARMV7，通常會跟在我們的 Library 後面。</p><figure name="6855" id="6855" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*wHa9bcDDiYCfGZ5AGxiWsA.png" data-width="1231" data-height="188" src="https://cdn-images-1.medium.com/max/800/1*wHa9bcDDiYCfGZ5AGxiWsA.png"></figure><p name="a88e" id="a88e" class="graf graf--p graf-after--figure">Crash Report 本來的內容僅會有給機器讀的語言，一大堆的記憶體位置，這時候我們需要 debug symbol 來做解譯的動作，我們稱為 Symbolicate ，而 Crash report 又依被解譯的程度分作 <br><strong class="markup--strong markup--p-strong"> — Not Symbolicated 尚未解譯<br> — Partially Symbolicated 部分解譯</strong><br>(通常是僅看到的 UIKit, Foundation 解譯的結果，因為在 MAC 上開啟 Crash Report 的時候會自動幫你載入) <br><strong class="markup--strong markup--p-strong"> — Fully Symbolicated 完全解譯</strong></p><p name="ad86" id="ad86" class="graf graf--p graf-after--p">那我們該如何進行解譯呢？<br>其實解譯的方法有很多種，例如透過 SymbolicateCrash (Xcode 內建)，atos (address to symbols)，或者你使用 Xcode 直接開啟等等都是不錯的方法，因此我這邊僅針對 atos 來做說明，因為工作上剛好有碰到，其他的工具等到我之後有碰到再來寫文章了QQ</p><p name="ba7f" id="ba7f" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">註: 如果是使用 Xcode 直接接上裝置，可以從 Window -&gt; Organizer 裡面，選到該支手機，並且點擊 View device logs 然後找到發生問題的那份 crash report，點擊右鍵就會有一個 re-symbolicate 的選項可以選擇了，從這邊會比較快速的找到問題點，但如果你的電腦上沒有特定 app 的 debug symbol 就不能用這招了。</em></p><p name="3a91" id="3a91" class="graf graf--p graf-after--p">打開 terminal，並且輸入下列指令</p><pre name="29ab" id="29ab" class="graf graf--pre graf-after--p">atos -arch {platform} -o {appname.dSYM} -l {load_address}</pre><p name="0b5f" id="0b5f" class="graf graf--p graf-after--pre">這邊我們先了解到幾個參數的用法</p><pre name="57f6" id="57f6" class="graf graf--pre graf-after--p"><br>-arch : 指定平台，就是剛剛提到的 ARM64/ARMV7<br>-o : 這邊需指定 app 的 Library<br>-l : 則是 binary 的 load address</pre><p name="be88" id="be88" class="graf graf--p graf-after--pre">輸入完之後，會發現很像進入了什麼 terminal 的畫面，這時我們就可以找到發生問題的 call stack 與其的記憶體位置，並且將其貼上去，他就會幫我們轉譯成我們看得懂的程式碼囉。</p><figure name="e027" id="e027" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*y-nEyw_SMH4yEVInOWdDfg.png" data-width="1912" data-height="175" src="https://cdn-images-1.medium.com/max/800/1*y-nEyw_SMH4yEVInOWdDfg.png"></figure><p name="bcc2" id="bcc2" class="graf graf--p graf-after--figure graf--trailing">好的，其實我很嘗試的去理解並且說明到讓我自己都覺得沒有問題的境界，但終究還是未夠班，如果有錯誤的地方歡迎指正。</p></div></div></section>
